# webgoat-labs

Практикум по DevSecOps/AppSec на базе **OWASP WebGoat** как внешнего учебного полигона.

> Исходный код WebGoat не хранится в этом репозитории. Вы клонируете его отдельно из официального источника.

## Формат модуля
- Ориентир по времени: **7 лабораторных по 20–30 минут**.
- Логика прохождения: **inventory → scan → analyze → prioritize → automate → gate → assess**.
- Фокус: не только запуск сканеров, но и **интерпретация результатов**, **triage** и **принятие решений**.

## Лабораторная 1: Подготовка WebGoat и карта security target
### Цель
Подготовить WebGoat к анализу и составить первичную карту цели: что именно будет сканироваться и почему.

### Шаги выполнения
1. Клонируйте WebGoat рядом с этим репозиторием.
2. Запустите WebGoat (через Docker или локально).
3. Проведите инвентаризацию проекта:
   - ключевые модули;
   - build-файлы;
   - dependency-файлы;
   - Dockerfile и контекст сборки;
   - директории, которые нужно сканировать в первую очередь.
4. Зафиксируйте **scope сканирования** (что входит/не входит в первую итерацию).

### Команды
```bash
git clone https://github.com/WebGoat/WebGoat.git
cd WebGoat

# Вариант 1: Docker
docker build -t webgoat-local .
docker run --rm -p 8080:8080 -p 9090:9090 webgoat-local

# Вариант 2: локально (если нужен Maven/JDK)
# ./mvnw clean install
# java -jar webgoat-server/target/webgoat-server-*.jar

# Инвентаризация структуры
find . -maxdepth 2 -type d
find . -maxdepth 2 -type f | head -n 80
```

### Ожидаемый результат
- WebGoat запущен локально.
- Сформирована первичная карта security target (модули + файлы + границы анализа).

### Артефакт
`webgoat-inventory.md`:
- 8–12 пунктов по структуре проекта;
- список активов для проверки (source, deps, image, secrets, SBOM);
- краткое обоснование, почему они в приоритете.

### Критерии проверки
- Есть рабочий запуск WebGoat.
- Есть документ с понятным scope первой итерации анализа.

### Вопросы на понимание
- Почему инвентаризация должна идти до запуска сканеров?
- Какие части проекта наиболее важны для SAST, а какие — для SCA?

## Лабораторная 2: SCA через Trivy FS с mini-prioritization
### Цель
Провести SCA-анализ зависимостей и научиться приоритизировать findings по риску, а не только по количеству.

### Шаги выполнения
1. Запустите `trivy fs` по WebGoat.
2. Сохраните JSON-отчёт.
3. Выделите топ-риски (например, 5 находок) по severity + контексту использования.
4. Подготовьте mini-prioritization: что исправлять первым, что можно отложить.

### Команды
```bash
cd WebGoat
trivy fs --scanners vuln --format json -o trivy-fs-report.json .
```

### Подсказки
- Смотрите не только на `CRITICAL/HIGH`, но и на:
  - наличие фиксированной версии;
  - тип зависимости (runtime/test);
  - потенциальную эксплуатируемость в вашем контексте.

### Ожидаемый результат
Есть не просто отчёт, а список приоритетов на исправление зависимостей.

### Артефакт
- `trivy-fs-report.json`
- `sca-prioritization.md` с 5 пунктами в формате:
  - `Component/CVE`
  - `Severity`
  - `Почему в приоритете`
  - `Предложение по remediation`

### Критерии проверки
- Отчёт сформирован.
- Приоритизация объясняет **почему** выбран такой порядок.

### Вопросы на понимание
- Почему “самый высокий severity” не всегда “самый срочный фикс”?
- Как отличать технический шум от реального риска для релиза?

## Лабораторная 3: Secret scanning через Gitleaks + политика исключений
### Цель
Проверить репозиторий на утечки секретов, отделить false positive от реальных рисков и предложить базовую policy для исключений.

### Шаги выполнения
1. Запустите `gitleaks detect` по директории проекта.
2. Сохраните JSON-отчёт.
3. Разберите findings и разметьте:
   - `true positive`;
   - `false positive`;
   - `needs context`.
4. Сформулируйте правила исключений (что и как можно игнорировать безопасно).

### Команды
```bash
cd WebGoat
gitleaks detect --source . --report-path gitleaks-report.json --report-format json
```

### Подсказки
- Проверяйте, является ли строка:
  - тестовым фикстурным значением;
  - реально используемым токеном/ключом;
  - случайным совпадением regex-паттерна.

### Ожидаемый результат
Подготовлен triage по секретам и минимальная policy исключений.

### Артефакт
- `gitleaks-report.json`
- `secret-triage.md`:
  - таблица `finding → classification → justification → action`;
  - 3–5 правил для исключений (например, для тестовых данных).

### Критерии проверки
- Есть обоснованное разделение true/false positives.
- Policy исключений не “отключает всё”, а снижает шум контролируемо.

### Вопросы на понимание
- Почему опасно без анализа массово добавлять ignore-правила?
- Что делать при обнаружении real secret в истории git?

## Лабораторная 4: SAST через OpenGrep с triage и remediation note
### Цель
Провести SAST-анализ и отработать практику triage: true positive, false positive, needs context, плюс приоритет исправлений.

### Шаги выполнения
1. Запустите OpenGrep по проекту и сохраните JSON.
2. Выберите минимум 8 findings для triage.
3. Для каждого finding укажите статус и краткое обоснование.
4. Для 3 наиболее важных true positive подготовьте remediation notes.

### Команды
```bash
cd WebGoat
opengrep --config auto --json --output opengrep-report.json .
```

### Ожидаемый результат
Сформирована triage-таблица и краткий план первичных исправлений.

### Артефакт
- `opengrep-report.json`
- `sast-triage.md`:
  - таблица `rule/file → status → severity → rationale`;
  - блок `Top-3 remediation notes` (что исправлять и как проверить фикс).

### Критерии проверки
- Есть минимум 8 разобранных findings.
- У каждого true positive есть понятный следующий шаг.

### Вопросы на понимание
- Почему часть findings помечается как `needs context`?
- Как связать SAST finding с реальным риском эксплуатации?

## Лабораторная 5: Container scanning и сравнение source vs image
### Цель
Сравнить результаты сканирования исходников и контейнерного образа, понять различия покрытий и ограничений каждого подхода.

### Шаги выполнения
1. Используйте уже собранный образ `webgoat-local` (не пересобирайте без необходимости).
2. Запустите `trivy image` и сохраните отчёт.
3. Сравните отчёт image scan с результатом `trivy fs` из Лабораторной 2.
4. Выделите:
   - что видно только в source scan;
   - что видно только в image scan;
   - почему для практики нужны оба типа проверки.

### Команды
```bash
cd WebGoat
trivy image --format json -o trivy-image-report.json webgoat-local
```

### Ожидаемый результат
Есть сравнительный анализ `fs vs image` и вывод по комбинированной стратегии сканирования.

### Артефакт
- `trivy-image-report.json`
- `source-vs-image-analysis.md`:
  - минимум 5 сравнительных наблюдений;
  - короткий вывод “какой scan на каком этапе SDLC полезнее”.

### Критерии проверки
- Сравнение основано на конкретных примерах из двух отчётов.
- Вывод объясняет, почему только одного типа scan недостаточно.

### Вопросы на понимание
- Какие риски появляются именно на уровне базового образа?
- Почему FS scan и image scan могут давать разный список CVE?

## Лабораторная 6: SBOM как инструмент supply chain и incident response
### Цель
Сгенерировать SBOM и использовать его не как “файл ради файла”, а как основу для анализа состава и сценариев реагирования.

### Шаги выполнения
1. Сгенерируйте SBOM для директории проекта и/или образа.
2. Выделите ключевые компоненты и их версии.
3. Опишите 2–3 сценария, где SBOM полезен:
   - быстрый поиск затронутых компонентов при новом CVE;
   - проверка состава перед релизом;
   - сверка с политикой допустимых компонентов.
4. Сопоставьте наблюдения с предыдущими SCA/container findings.

### Команды
```bash
cd WebGoat
syft dir:. -o spdx-json=sbom-webgoat.spdx.json
# при необходимости:
# syft webgoat-local -o spdx-json=sbom-webgoat-image.spdx.json
```

### Ожидаемый результат
SBOM интерпретирован как рабочий артефакт supply chain, а не как формальная выгрузка.

### Артефакт
- `sbom-webgoat.spdx.json` (или `sbom-webgoat-image.spdx.json`)
- `sbom-interpretation.md`:
  - 5–7 наблюдений по составу;
  - 2–3 практических use case для incident response.

### Критерии проверки
- Есть связь между SBOM и реальными задачами управления риском.
- Интерпретация опирается на конкретные компоненты из SBOM.

### Вопросы на понимание
- Почему SBOM важен для прозрачности цепочки поставки?
- Чем SBOM дополняет SCA-скан, а не заменяет его?

## Лабораторная 7: Локальный security pipeline + quality gate + мини-ассессмент
### Цель
Собрать результаты в единый workflow: автоматизированный запуск сканеров, сравнение режимов `warn-only` и `quality gate`, финальный mini security assessment WebGoat.

### Шаги выполнения
1. Запустите локальный pipeline в режиме `warn-only`.
2. Запустите pipeline в режиме `quality gate`.
3. Сравните финальные статусы и `exit code`.
4. Подготовьте мини-ассессмент на основе всех лабораторных:
   - top risks;
   - quick wins;
   - улучшения процесса.

### Команды
```bash
WARN_ONLY=true FAIL_ON_ANY_ERROR=false bash ../devsecops-intern-labs/scanner-scripts/full_local_pipeline.sh . webgoat-local . | tee pipeline-warn-output.txt
WARN_ONLY=false FAIL_ON_ANY_ERROR=true bash ../devsecops-intern-labs/scanner-scripts/full_local_pipeline.sh . webgoat-local . | tee pipeline-gate-output.txt
```

### Подсказки
- Зафиксируйте, какие проверки влияют на блокирующий статус.
- В финальном выводе разделите:
  - **технические уязвимости**;
  - **операционные улучшения процесса** (политики, автоматизация, triage-практика).

### Ожидаемый результат
Понимание, как один и тот же набор сканов ведёт себя в двух режимах, и итоговый assessment как мини-результат “под релиз”.

### Артефакт
- `pipeline-warn-output.txt`
- `pipeline-gate-output.txt`
- `pipeline-run-summary.md` (статусы, exit codes, выводы)
- `webgoat-mini-security-assessment.md`:
  - mini risk register (5–8 рисков);
  - приоритизация `P1/P2/P3`;
  - remediation backlog на ближайшие итерации.

### Критерии проверки
- Продемонстрирована разница `warn-only` vs `quality gate` на фактических выводах.
- Мини-ассессмент содержит не только проблемы, но и план действий.

### Вопросы на понимание
- Почему quality gate должен быть настраиваемым, а не “всегда блокирующим”?
- Какие риски стоит закрывать в первую очередь на уровне intern/junior команды?

## Итог модуля
После прохождения блока студент должен уметь:
- анализировать структуру реального security target;
- запускать и сравнивать несколько типов сканирования (SCA, secrets, SAST, container);
- делать базовый triage и приоритизацию findings;
- использовать SBOM для задач supply chain/incident response;
- запускать локальный security pipeline и объяснять логику quality gate;
- оформлять короткий security assessment с понятными артефактами.
