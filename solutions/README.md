# solutions

Раздел для преподавателя и самопроверки. Ответы даны кратко, без полного раскрытия всех деталей, чтобы сохранить учебную ценность.

## 1) Краткие эталонные ответы по лабораторным

### vulnerable-app
- `/health` — безопасный endpoint: нет пользовательского ввода, минимальная поверхность атаки.
- `/diagnostic/ping` — учебный анти-паттерн: использование `shell=True` и пользовательского ввода в командной строке.
- `/calc` — риск недостаточной валидации и отсутствия ограничений диапазона.

### docker-labs
- Основные anti-patterns: root-пользователь, нефиксированные теги, избыточные пакеты.
- Базовые меры: non-root user, slim base image, pinning версий, минимизация layers.

### webgoat-labs
- Инструменты дают сигнал, но final verdict требует контекста проекта.
- Важны triage, приоритизация и план устранения.

## 2) Пример хорошего Dockerfile

```dockerfile
FROM python:3.11-slim

WORKDIR /app

RUN adduser --disabled-password --gecos "" appuser

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY app.py .

USER appuser
EXPOSE 5000

CMD ["python", "app.py"]
```

## 4) Пример краткого remediation plan

1. Убрать `shell=True` и заменить безопасным вызовом subprocess с аргументами списком.
2. Ужесточить валидацию пользовательского ввода (формат, длина, allow-list).
3. Пересобрать образ на минимальной базе и добавить non-root user.
4. Обновить уязвимые зависимости до безопасных версий.
5. Повторно выполнить сканы и сверить снижение риска.

## 5) Интерпретация результатов инструментов

- **Trivy:** смотрим severity + fix availability; приоритизируем HIGH/CRITICAL с доступным патчем.
- **Gitleaks:** валидируем каждое срабатывание, отделяя тестовые строки от реальных секретов.
- **OpenGrep:** разделяем true/false positives, проверяем потенциальный exploit path.
- **Syft:** используем SBOM как inventory компонентов и базу для дальнейшего risk management.
