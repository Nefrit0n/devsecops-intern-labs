## Почему секреты — отдельный процесс

ГОСТ Р 56939-2024 впервые выделил безопасность секретов в **отдельный процесс (5.15)** — раньше этого не было. Причина простая: реальные инциденты.

В 2022 году Uber был взломан через захардкоженные credentials, найденные в репозитории. Каждый день на GitHub появляются тысячи случайно закоммиченных API-ключей — боты сканируют их и эксплуатируют за минуты.

**Важный нюанс:** `git rm secret.txt` НЕ удаляет секрет. Он остаётся в Git history навсегда. Именно поэтому нужны инструменты, сканирующие *всю историю*, а не только текущую версию.

---

## Три философии поиска секретов

| Инструмент | Философия | Когда использовать |
|------------|-----------|-------------------|
| **Gitleaks** | «Поставь на ворота — не пропускай секреты в репо» | Pre-commit hook, CI pipeline |
| **TruffleHog** | «Просканируй всё и проверь, что ключ ещё работает» | Аудит, incident response |
| **detect-secrets** | «Зафиксируй текущее, лови только новое» | Легаси-кодовая база с историческим мусором |

Идеальная комбинация: **Gitleaks на pre-commit** (предотвращение) + **TruffleHog в CI** (обнаружение + верификация) + **detect-secrets baseline** (для легаси).
