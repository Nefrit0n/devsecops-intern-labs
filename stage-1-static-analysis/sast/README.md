## Что такое SAST (и чем он не является)

SAST (**S**tatic **A**pplication **S**ecurity **T**esting) анализирует код *без его запуска*. Представьте, что вы читаете рецепт и замечаете «добавьте 500 г цианида» — вам не нужно готовить блюдо, чтобы понять, что что-то не так.

**Что SAST находит хорошо:**
- Паттерны небезопасного кода (SQL-конкатенация, `eval`, `exec`)
- Использование устаревших/опасных функций (`md5`, `pickle.loads`)
- Нарушения правил кодирования
- Захардкоженные значения

**Что SAST находит плохо:**
- Проблемы бизнес-логики (IDOR, broken access control)
- Уязвимости, проявляющиеся только в runtime
- Проблемы конфигурации запущенного сервиса

Именно поэтому SAST — не замена DAST (этап 3), а дополнение. Вместе они покрывают больше.

## Три инструмента — три подхода

| Инструмент | Как анализирует | Аналогия |
|------------|----------------|----------|
| **Semgrep** | Pattern matching по AST | «Найди все места, где код выглядит *вот так*» |
| **Bandit** | Python AST + плагины | «Я знаю 100 опасных Python-паттернов наизусть» |
| **njsscan** | Semgrep + кастомные Node.js правила | «Я специализируюсь на Node.js и знаю его слабости» |

**Почему три, а не один?** Потому что каждый видит разное:
- Semgrep найдёт SQL injection в JS, но пропустит Python-специфичный `pickle.loads`
- Bandit поймает `pickle.loads`, но не знает что такое `express`
- njsscan найдёт `child_process.exec(userInput)` в Node.js, который Semgrep может пропустить с дефолтными правилами

На этой лабе вы прогоните один и тот же Juice Shop через все три — и увидите разницу.
