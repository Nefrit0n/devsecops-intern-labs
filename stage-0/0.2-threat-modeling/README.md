## Зачем моделировать угрозы

Представьте: вы охраняете здание. Можно поставить камеры везде (дорого, половина смотрит в стену). А можно сначала **нарисовать план здания**, отметить входы, окна, ценности внутри — и поставить камеры только туда, где реальный риск.

Моделирование угроз — это «план здания» для вашего приложения. Без него вы либо сканируете всё подряд (дорого и шумно), либо пропускаете критичные точки.

**ГОСТ Р 56939-2024, раздел 5.7** прямо говорит: *определите поверхность атаки до начала исследований кода*. Сначала карта — потом разведка.

## Два инструмента, которые будем использовать

### DFD (Data Flow Diagram)

DFD отвечает на вопрос: **«Как данные двигаются через систему?»**

Четыре элемента:

| Элемент | Форма | Что это | Пример в Juice Shop |
|---------|-------|---------|---------------------|
| Процесс | ⭕ круг | Код, который обрабатывает данные | Backend (Express.js) |
| Хранилище | ═══ параллельные линии | Где данные лежат | SQLite database |
| Внешняя сущность | ▭ прямоугольник | Всё, что вне нашего контроля | Пользователь, Payment API |
| Поток данных | → стрелка | Данные в движении | HTTP запрос, SQL-запрос |

И самый важный элемент — **Trust Boundary** (граница доверия): пунктирная линия, разделяющая зоны с разным уровнем доверия.

**Правило:** атаки происходят на границах доверия. Если данные пересекают trust boundary — это потенциальная точка атаки.

### STRIDE

STRIDE отвечает на вопрос: **«Что может пойти не так на каждой границе?»**

| Буква | Угроза | Мнемоника | Типичная атака |
|-------|--------|-----------|----------------|
| **S** | Spoofing | **С**может ли враг притвориться кем-то другим? | Кража сессии, подделка JWT |
| **T** | Tampering | **Т**рогал ли кто-то данные по дороге? | Модификация запроса, SQL injection |
| **R** | Repudiation | Сможет ли пользователь сказать «это не я»? | Отсутствие audit log |
| **I** | Info Disclosure | Может ли **И**нформация утечь? | Ошибки с traceback, directory listing |
| **D** | Denial of Service | Можно ли **Д**обить сервис? | ReDoS, бесконечные запросы |
| **E** | Elevation | Может ли юзер стать админом? | IDOR, broken access control |

## Поверхность атаки (Attack Surface)

Прежде чем рисовать DFD, определите **поверхность атаки** — все точки, через которые внешний мир взаимодействует с приложением.

Для Juice Shop это:
- Каждый HTTP-эндпоинт (API + страницы)
- Формы пользовательского ввода
- Загрузка файлов
- WebSocket-соединения
- Cookies и заголовки
- Зависимости (npm-пакеты)

**Задание по определению поверхности атаки — в tasks.md.**

## Архитектура Juice Shop (для справки)

```
                     ┌──────────────────────────────┐
   Пользователь      │      Trust Boundary: DMZ      │
   (браузер)  ──────►│                                │
                     │  ┌────────────┐  REST API  ┌──────────────┐
         HTTP/       │  │  Angular   │ ─────────► │  Express.js  │
         WebSocket   │  │  Frontend  │ ◄───────── │   Backend    │
                     │  └────────────┘   JSON     │              │
                     │                             │   Sequelize  │
                     │                             └──────┬───────┘
                     │                                    │ SQL
                     │  ┌─────────────────────────────────┤
                     │  │      Trust Boundary: Data       │
                     │  │                                 │
                     │  │  ┌──────────┐  ┌─────────────┐ │
                     │  │  │  SQLite  │  │ File System  │ │
                     │  │  │   (DB)   │  │  (uploads)   │ │
                     │  │  └──────────┘  └─────────────┘ │
                     │  └─────────────────────────────────┘
                     └──────────────────────────────────────┘
```

**Trust boundaries:**
1. Интернет ↔ Frontend (пользовательский ввод не доверенный)
2. Frontend ↔ Backend API (фронт может быть подделан)
3. Backend ↔ Database (SQL-запросы могут содержать инъекции)
4. Backend ↔ File System (загруженные файлы могут быть вредоносными)
