## Задание 1 · Поверхность атаки Juice Shop
**Тег:** 🟢 практика · **Время:** ~1 ч

**Перед тем как рисовать DFD** — определите поверхность атаки.

1. Запустите Juice Shop: `docker run --rm -p 3000:3000 bkimminich/juice-shop`
2. Откройте http://localhost:3000
3. Пройдитесь по приложению, откройте DevTools → Network
4. Заполните шаблон [`templates/attack-surface-template.md`](templates/attack-surface-template.md):
   - Все HTTP-эндпоинты, которые увидите в Network tab
   - Все формы ввода (регистрация, поиск, отзывы, оплата)
   - Загрузка файлов (если есть)
   - Cookies и заголовки авторизации

**Подсказка:** в Juice Shop есть скрытые эндпоинты. Попробуйте `/api-docs` или посмотрите в исходном коде.

> **Зачем:** ГОСТ 5.7 требует определить поверхность атаки *до* моделирования угроз. Это не формальность — без неё DFD будет неполной.

## Задание 2 · DFD Level 0 и Level 1
**Тег:** 🟢 практика · **Время:** ~1.5 ч

Постройте две диаграммы:

**Level 0** (контекстная) — одна «коробка» Juice Shop, вокруг — внешние сущности:
- Пользователь (браузер)
- Внешние API (если есть)
- Администратор

**Level 1** (детальная) — раскрываем «коробку»:
- Frontend (Angular SPA)
- Backend API (Express.js)
- Database (SQLite)
- File system (uploads)
- Обязательно: **минимум 4 trust boundaries**

Для каждого потока данных подпишите:
- Что передаётся (HTTP request, JSON, SQL query, file)
- Протокол (HTTP, WebSocket, SQL)
- Направление (→ или ↔)

**Инструмент:** draw.io (File → New → Blank Diagram)

**Совет:** не пытайтесь нарисовать всё идеально с первого раза. Начните с 3–4 основных компонентов, потом добавляйте детали.

## Задание 3 · STRIDE-анализ
**Тег:** 🟢 практика · **Время:** ~1.5 ч

Для **каждого потока данных, пересекающего trust boundary** на вашей DFD — пройдите по STRIDE:

```markdown
| # | Поток (trust boundary) | S | T | R | I | D | E | Сценарий атаки | Severity | Мера |
|---|------------------------|---|---|---|---|---|---|----------------|----------|------|
```

**Правила:**
- Не ставьте ✓ во все ячейки — подумайте, какие угрозы реально применимы
- Сценарий атаки — **конкретный**, не абстрактный. Не «может быть XSS», а «злоумышленник вставляет `<script>` в поле отзыва, скрипт исполняется у других пользователей»
- Severity: **HIGH** = утечка данных / полный доступ, **MEDIUM** = частичный ущерб, **LOW** = сложная эксплуатация или малый ущерб
- Мера — что конкретно нужно сделать (не «улучшить безопасность», а «добавить Content-Security-Policy header»)

**Шаблон:** [`templates/stride-template.md`](templates/stride-template.md)

> **Подсказка:** начните с потоков User → Frontend и Frontend → Backend — там больше всего угроз. Backend → DB — это почти всегда T (SQL injection) и I (утечка через error messages).

## Задание 4 · pytm — модель как код
**Тег:** 🟢 практика · **Время:** ~1 ч

Зачем pytm, если есть draw.io? Потому что модель угроз, описанная кодом:
- Версионируется в Git
- Ревьюится в pull request
- Автоматически генерирует отчёт
- Может запускаться в CI/CD

```bash
pip install pytm
```

1. Возьмите пример [`examples/pytm-example.py`](examples/pytm-example.py)
2. Адаптируйте его под вашу DFD Juice Shop
3. Запустите:

```bash
# Генерация DFD
python threat-model.py --dfd | dot -Tpng -o dfd-juice-shop.png

# Генерация отчёта
python threat-model.py --report pytm-report.md
```

4. **Сравните** автоматический отчёт pytm с вашим ручным STRIDE:
   - Что pytm нашёл, а вы пропустили?
   - Что вы нашли, а pytm пропустил? (Подсказка: pytm не понимает бизнес-логику)
   - Запишите выводы в конец файла `stride-analysis.md`
