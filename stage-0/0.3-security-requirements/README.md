# Модуль 0.3 · Требования безопасности

**Время:** ~3 часа · **Тип:** 🟢 практика
**Привязка к ГОСТу:** раздел 5.6 — Формирование требований безопасности к ПО

---

## Зачем нужны требования (а не просто «запустить сканер»)

Вот типичный диалог без требований безопасности:

> — Сканер нашёл 87 findings. Мы можем релизить?
> — Не знаю. А какие из них критичные?
> — Не знаю. Может, все?
> — Ладно, просто исправим что успеем…

А вот с требованиями:

> — Сканер нашёл 87 findings. Из них 3 нарушают наши security requirements с severity HIGH.
> — Какие?
> — T-01: SQL-запрос без параметризации в `/api/Users`. I-03: stack trace в ответе API. S-02: JWT без проверки срока действия.
> — Это блокеры. Остальные 84 — в бэклог. Не релизим, пока не починим эти три.

Требования превращают хаос в систему. Они становятся **quality gates** в пайплайне на этапе 5.

---

## Что такое хорошее требование

**Плохо** (не проверяемо):
> «Приложение должно быть безопасным»
> «Необходимо обеспечить защиту от XSS»

**Хорошо** (проверяемо):
> «Backend НЕ ДОЛЖЕН возвращать stack trace в HTTP-ответах с кодом 4xx/5xx» — проверяется DAST-сканером
> «Все SQL-запросы ДОЛЖНЫ использовать параметризованные выражения (prepared statements)» — проверяется SAST-правилом Semgrep

**Формула:** `[Компонент] + [ДОЛЖЕН/НЕ ДОЛЖЕН] + [конкретное действие] + [как проверить]`

## Откуда брать требования

Ваш главный источник — **STRIDE-таблица из модуля 0.2**. Каждая найденная угроза порождает одно или несколько требований:

```
Угроза (STRIDE):
  T-3: SQL injection в /api/Users (severity HIGH)
      ↓
Требование:
  T-01: Backend ДОЛЖЕН использовать Sequelize ORM с параметризованными запросами
        для всех обращений к базе данных.
  Проверка: Semgrep rule `javascript.sequelize.security.audit`
  Этап: CI (SAST)
  Pass/Fail: 0 findings категории ERROR
```

---

## Трассируемость — ключевое слово

Аудитор по ГОСТу спросит не просто «есть ли требования», а **«можете ли вы проследить цепочку от угрозы до проверки»**:

```
Угроза (5.7) → Требование (5.6) → Инструмент проверки (5.9/5.11) → Quality gate (5.14)
```

Если цепочка разрывается — процесс не работает. Именно поэтому в шаблоне есть колонки «Угроза», «Инструмент», «Этап», «Pass/Fail».
