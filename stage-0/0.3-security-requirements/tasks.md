# Задания · Модуль 0.3

---

## Задание 1 · Формирование требований из STRIDE
**Тег:** 🟢 практика · **Время:** ~1.5 ч

Откройте вашу STRIDE-таблицу из модуля 0.2. Для каждой угрозы с severity **HIGH** или **MEDIUM**:

1. Сформулируйте требование по формуле: `[Компонент] ДОЛЖЕН/НЕ ДОЛЖЕН [действие]`
2. Требование должно быть **проверяемым инструментом**, а не экспертной оценкой

**Минимум 15 требований**, сгруппированных по STRIDE-категориям.

**Шаблон:** [`templates/requirements-template.md`](templates/requirements-template.md)

Примеры хороших требований:

| ID   | Требование | Угроза | Инструмент |
|------|-----------|--------|------------|
| S-01 | Backend ДОЛЖЕН проверять JWT-подпись на каждом защищённом эндпоинте | STRIDE #1 | Semgrep + DAST (ZAP auth scan) |
| T-01 | Все SQL-запросы ДОЛЖНЫ использовать параметризованные выражения | STRIDE #3 | Semgrep rule `sql-injection` |
| T-02 | Frontend НЕ ДОЛЖЕН вставлять пользовательский ввод в DOM без санитизации | STRIDE #5 | njsscan + ZAP XSS scan |
| I-01 | Backend НЕ ДОЛЖЕН возвращать stack trace в production | STRIDE #7 | DAST (ZAP error scan) |
| I-02 | Исходный код НЕ ДОЛЖЕН содержать захардкоженные пароли и ключи | STRIDE #9 | Gitleaks |

## Задание 2 · Трассировка к пайплайну
**Тег:** 🟢 практика · **Время:** ~1 ч

Для каждого требования из задания 1 заполните расширенную таблицу:

```markdown
| ID   | Требование | Угроза | Инструмент | Конкретное правило/конфиг | Этап CI/CD | Критерий pass/fail |
|------|-----------|--------|------------|--------------------------|------------|-------------------|
| T-01 | Параметризованные SQL | #3 | Semgrep | `rules/sql-injection.yml` | pre-commit + CI | 0 findings HIGH |
```

Новая колонка — **«Конкретное правило/конфиг»**. Это не просто «Semgrep», а *какое именно правило*. На этапе 1 вы создадите эти конфиги, а на этапе 5 — встроите их в пайплайн.

---

## Задание 3 · Приоритизация
**Тег:** 🟢 практика · **Время:** ~30 мин

Не все требования одинаково важны. Разделите ваши 15+ требований на три уровня:

- 🔴 **Блокер** — без выполнения этого требования релизить нельзя (quality gate = fail)
- 🟡 **Важно** — нужно исправить до следующего релиза (warning в пайплайне)
- 🟢 **Рекомендация** — хорошо бы, но не блокирует (info в отчёте)

Добавьте колонку «Приоритет» в таблицу требований.

> **Совет:** блокеров не должно быть 15 из 15. Если всё — блокер, то ничего не блокер. Типично: 3–5 блокеров, 5–7 важных, остальное — рекомендации.

---

## Задание 4 · Проверка полноты
**Тег:** 🟣 теория · **Время:** ~20 мин

Пройдите по чеклисту:

- [ ] Есть хотя бы 1 требование для **каждой** STRIDE-категории (S, T, R, I, D, E)
- [ ] Для каждого требования указан **конкретный инструмент** (не «какой-нибудь SAST»)
- [ ] Каждое требование **трассируется** к конкретной угрозе из STRIDE-таблицы
- [ ] Есть **критерий pass/fail** (не «исправить по возможности», а «0 findings HIGH»)
- [ ] Приоритеты расставлены (не всё — блокер)

Если какой-то пункт не выполнен — доработайте перед тем как двигаться дальше. Этот документ — основа всего курса.
