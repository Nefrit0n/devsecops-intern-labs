# Модуль 0.1 · ГОСТ: из канцелярита в человеческий

**Время:** ~4 часа · **Тип:** 🟣 теория + 🟢 практика

---

## Что за зверь — ГОСТ Р 56939-2024

Если коротко: это документ, который говорит разработчикам ПО **«вот как надо строить процессы, чтобы ваш софт не взломали, а если взломают — вы хотя бы быстро узнаете и почините»**.

Полное название: *«Защита информации. Разработка безопасного программного обеспечения. Общие требования»*.

Пришёл на смену версии 2016 года. Вступил в силу 20 декабря 2024.

**Кто написал:** ФСТЭК + Лаборатория Касперского + ИСП РАН + Positive Technologies + InfoTeCS + РусБИТех-Астра + СберТех + ещё несколько организаций. То есть не один чиновник в кабинете, а реальные разработчики СЗИ.

---

## Кому это надо (и зачем вам)

**Обязательно:**
- Разработчикам средств защиты информации (приказ ФСТЭК № 240 — сертификация процессов РБПО)
- Разработчикам ПО для ГИС (приказ ФСТЭК № 117)
- Разработчикам ПО для объектов КИИ (приказ ФСТЭК № 159)

**Полезно:**
- Любой компании, которая хочет повысить зрелость безопасной разработки
- Финтеху, ритейлу, телекому — везде, где утечка данных = репутационные и финансовые потери

**Вам лично:**
- Если вы DevSecOps — вы будете внедрять эти процессы
- Если вы AppSec — вы будете проверять их выполнение
- Если вы разработчик — вы будете жить внутри них

---

## Главное отличие: мера ≠ процесс

| Версия 2016 (старая)                        | Версия 2024 (новая)                           |
|----------------------------------------------|------------------------------------------------|
| «Проводите статический анализ» (мера)        | «Выстройте процесс статического анализа» (процесс) |
| Сделал один раз — ок                         | Должен быть повторяемый, документируемый       |
| Не описывает что предъявить аудитору         | Чёткие требования к артефактам (свидетельствам) |
| ~14 мер                                      | 25 процессов (8 новых)                         |

**Аналогия:** версия 2016 говорила «мойте руки». Версия 2024 говорит «постройте систему, в которой есть раковина, мыло, инструкция, контроль и отчёт о том, что руки были вымыты».

## 25 процессов: разложим по полочкам

Не надо зубрить 25 пунктов. Они логически группируются в **5 блоков** — как этажи здания. Запомните блоки, внутри каждого 4–6 процессов.

### 🏗️ Блок 1 · Фундамент (управление)

*Прежде чем что-то делать — спланируй, обеспечь ресурсами, обучи людей.*

| Раздел | Процесс | Человеческим языком | Что будет если забить |
|--------|---------|--------------------|-----------------------|
| 5.1 | Планирование процессов РБПО | «Какие процессы нам нужны, кто отвечает, какие ресурсы» | Хаос. Каждый делает что-то своё, ничего не документировано |
| 5.2 | Обеспечение инструментальной среды | «Какие инструменты используем, как настраиваем» | У каждого свой сканер с разными настройками, результаты несравнимы |
| 5.3 | Управление ИТ-инфраструктурой | «Среда разработки тоже должна быть безопасной» | Разработчик сидит на заражённой машине → supply chain attack |
| 5.4 | Управление конфигурацией ПО | «Git, версионирование, контроль изменений» | Кто-то закоммитил в main без review → уязвимость в продакшене |
| 5.5 | Обучение сотрудников | «Команда должна знать, что такое OWASP Top 10» | Разработчик не знает про SQL injection → пишет `SELECT * WHERE id = ' + userInput` |

### 📋 Блок 2 · Проектирование (до кода)

*Определи что защищаем и от чего — до того как напишешь первую строку.*

| Раздел | Процесс | Человеческим языком | Что будет если забить |
|--------|---------|--------------------|-----------------------|
| 5.6 | Формирование требований безопасности | «Чеклист: что приложение ДОЛЖНО и НЕ ДОЛЖНО делать» | Нет критериев приёмки → сканер нашёл 100 findings, непонятно, это критично или нет |
| 5.7 | Моделирование угроз | «Нарисуй схему, определи где враг может атаковать» | Защитили фронтенд, забыли про API → взломали через бэкдор |
| 5.8 | Правила кодирования | «Единые стандарты: как писать безопасный код» | Один пишет prepared statements, другой конкатенирует строки в SQL |

### 🔬 Блок 3 · Тестирование (ищем баги)

*Код написан — теперь ищем в нём дыры всеми возможными способами.*

| Раздел | Процесс | Человеческим языком | Что будет если забить |
|--------|---------|--------------------|-----------------------|
| 5.9 | Статический анализ (SAST) | «Сканируем код, не запуская его» | Пропускаем очевидные паттерны: hardcoded пароли, небезопасные функции |
| 5.10 | Композиционный анализ (SCA) | «Проверяем чужие библиотеки на CVE» | log4shell в зависимостях → взлом через стороннюю библиотеку |
| 5.11 | Динамический анализ (DAST) | «Сканируем запущенное приложение снаружи» | SAST не нашёл, потому что уязвимость проявляется только в runtime |
| 5.12 | Функциональное тестирование | «Проверяем: аутентификация работает? RBAC работает?» | Логин работает, но можно обойти проверку прав и стать админом |
| 5.13 | Нефункциональное тестирование | «Пентест, фаззинг, нагрузочное» | Все юнит-тесты зелёные, но приложение падает от 100 запросов в секунду |
| 5.19 | Экспертиза исходного кода | «Ручной code review с фокусом на безопасность» | Автосканер не понимает бизнес-логику → пропускает IDOR |

### 🔒 Блок 4 · Защита процесса (секреты, релизы, конфигурация)

*Не только код, но и всё вокруг него должно быть защищено.*

| Раздел | Процесс | Человеческим языком | Что будет если забить |
|--------|---------|--------------------|-----------------------|
| 5.14 | Управление недостатками | «Трекинг уязвимостей: нашли → зафиксировали → проверили → закрыли» | Нашли критичную уязвимость, забыли, через 3 месяца взломали |
| 5.15 | Безопасность секретов | «Пароли, ключи, токены — не в коде, не в .env» | API-ключ в GitHub → бот его сканирует → ваш AWS-аккаунт майнит крипту |
| 5.16 | Безопасность при выпуске | «Подписание артефактов, проверка целостности» | Кто-то подменил binary между CI и продакшеном → supply chain attack |

### 🔄 Блок 5 · Жизненный цикл (после релиза)

*Выпустили — теперь следите, обновляйте, и однажды аккуратно выключите.*

| Раздел | Процесс | Человеческим языком | Что будет если забить |
|--------|---------|--------------------|-----------------------|
| 5.17 | Безопасность при эксплуатации | «Мониторинг, реагирование на инциденты, патчинг» | Уязвимость в проде, узнаёте через 6 месяцев из новостей |
| 5.18 | Безопасность при выводе из эксплуатации | «Удалить данные, отозвать ключи, уведомить пользователей» | Выключили сервер, забыли про БД с персданными на том же диске |
| 5.20 | Управление документацией | «Документация по процессам РБПО» | Аудитор приходит — нечего показать |
| 5.21 | Управление изменениями | «Контроль изменений в коде и процессах» | Кто-то тихо поменял конфиг → сломал безопасность, никто не заметил |
| 5.22 | Управление рисками | «Оценка и приоритизация рисков» | Всё одинаково важно = ничего не важно |
| 5.23 | Управление поставщиками | «Контроль безопасности внешних подрядчиков» | Аутсорсер пишет код без review → уязвимости в продакшене |
| 5.24 | Обеспечение осведомлённости | «Информирование пользователей об угрозах» | Пользователи не знают про 2FA → массовый фишинг |
| 5.25 | Совершенствование процессов | «Ретроспектива: что улучшить в РБПО» | Те же грабли каждый релиз |

## Мнемоника: 5 блоков = 5 пальцев

Загните пальцы одной руки:

1. 🏗️ **Фундамент** — планируем, обеспечиваем, обучаем (5 процессов)
2. 📋 **Проектирование** — требования, угрозы, правила кода (3 процесса)
3. 🔬 **Тестирование** — SAST, SCA, DAST, пентест, review (6 процессов)
4. 🔒 **Защита процесса** — секреты, баг-трекинг, релизы (3 процесса)
5. 🔄 **Жизненный цикл** — эксплуатация, документация, улучшение (8 процессов)

**5 + 3 + 6 + 3 + 8 = 25 процессов.** Запомнили? Проверьте себя.

## Что нового в версии 2024 (8 процессов)

Эти процессы *не существовали* в версии 2016. Их добавили, потому что индустрия за 8 лет натолкнулась на реальные проблемы:

| Новый процесс | Почему добавили | Реальный кейс |
|---------------|-----------------|---------------|
| 5.1 Планирование РБПО | Без плана внедрение буксовало | «Мы внедрили SAST, но забыли про людей и бюджет» |
| 5.3 Управление инфраструктурой | SolarWinds показал: среда разработки — тоже вектор | SolarWinds Orion (2020) — заражение через build-систему |
| 5.10 Композиционный анализ (SCA) | 80%+ кода — сторонние библиотеки | Log4Shell (2021) — 0-day в зависимости |
| 5.15 Безопасность секретов | Массовые утечки ключей через Git | Uber (2022) — hardcoded credentials в репозитории |
| 5.16 Безопасность при выпуске | Supply chain attacks растут | Codecov (2021) — подмена артефакта в CI |
| 5.17 Безопасность при эксплуатации | Раньше ГОСТ заканчивался на релизе | Equifax (2017) — не запатчили известную CVE |
| 5.18 Вывод из эксплуатации | Данные оставались на мёртвых серверах | Утечки из «выключенных» сервисов |
| 5.23 Управление поставщиками | Аутсорс без контроля = дыра | Kaseya (2021) — атака через вендора MSP |
