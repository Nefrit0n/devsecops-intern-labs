# Задания · Модуль 0.1

---

## Задание 1 · Самопроверка: 5 блоков
**Тег:** 🟣 теория · **Время:** ~20 мин

Закройте README модуля. По памяти напишите названия 5 блоков процессов ГОСТа и сколько процессов в каждом. Затем откройте и проверьте.

Цель — не 100% точность, а понимание структуры. Если помните 4 из 5 блоков — уже хорошо.

---

## Задание 2 · Маппинг процессов → инструменты → артефакты
**Тег:** 🟢 практика · **Время:** ~1.5 ч

Откройте ГОСТ Р 56939-2024 (раздел 5). Создайте файл `gost-mapping.md` и заполните таблицу:

```markdown
| Блок | № | Процесс | Раздел | Инструмент (open-source) | Артефакт для аудитора | Наш этап |
|------|---|---------|--------|--------------------------|----------------------|----------|
| 🏗️  | 1 | Планирование РБПО | 5.1 | Jira | План РБПО | 0 |
| …    |   |         |        |                          |                      |          |
```

**Заполните все 25 строк.** Подсказки:
- Колонка «Инструмент» — чем можно закрыть процесс. Если процесс организационный (типа обучения) — напишите инструмент управления (Confluence, Notion)
- Колонка «Артефакт» — что покажете аудитору. Это может быть отчёт сканера, документ, конфигурация CI/CD
- Колонка «Наш этап» — к какому этапу курса (0–5) относится процесс

> **Зачем:** это упражнение заставляет прочитать каждый подраздел ГОСТа и подумать «а как это выглядит на практике». После него 25 процессов перестают быть абстракцией.

## Задание 3 · 8 новых: почему их добавили
**Тег:** 🟢 практика · **Время:** ~1 ч

Для каждого из 8 новых процессов (которых не было в 2016) напишите:

1. **Что это** (1 предложение своими словами, не копипаста из ГОСТа)
2. **Реальный инцидент**, который показал зачем это нужно (из таблицы в README или найдите свой)
3. **Как бы этот процесс помог** предотвратить или смягчить инцидент (1–2 предложения)

Добавьте в конец `gost-mapping.md` под заголовком `## Новые процессы 2024`.

> **Зачем:** привязка к реальным инцидентам — лучший способ запомнить. «Безопасность секретов» — абстракция. «Uber взломали через hardcoded credentials в репозитории» — запоминается навсегда.

## Задание 4 · Шпаргалка
**Тег:** 🟢 практика · **Время:** ~40 мин

Создайте файл `gost-cheatsheet.md` — вашу личную шпаргалку по ГОСТу. Формат свободный, но вот требования:

- Уместить все 25 процессов **на одном экране** (без скролла)
- Для каждого процесса — **максимум 5 слов** описания
- Группировка по 5 блокам
- Должна быть полезной через месяц, когда вы забудете детали

Пример формата:

```
🏗️ ФУНДАМЕНТ
  5.1 План          — кто, что, когда, бюджет
  5.2 Инструменты   — какие тулы, как настроены
  5.3 Инфраструктура— безопасность среды разработки
  5.4 Конфигурация  — git, версии, контроль изменений
  5.5 Обучение      — команда знает OWASP Top 10
```

> **Зачем:** процесс создания шпаргалки — это и есть запоминание. Вы вынуждены переформулировать каждый процесс в 5 слов, а значит — понять его суть.

## Задание 5 · Квиз: «что сломалось?»
**Тег:** 🟣 теория · **Время:** ~30 мин

Прочитайте каждый сценарий и определите, какой процесс ГОСТа был нарушен. Может быть несколько ответов.

**Сценарий A:**
Команда выпустила релиз. Через неделю на Хабре появляется статья: в вашем приложении SQL-инъекция в форме поиска. Вы не знали, потому что поисковый модуль писал аутсорсер, и его код никто не проверял.

<details>
<summary>Ответ</summary>

- **5.23** Управление поставщиками — код аутсорсера не проверялся
- **5.9** Статический анализ — SAST не запускался на коде аутсорсера
- **5.19** Экспертиза кода — не было code review
- **5.7** Моделирование угроз — форма поиска не была в модели угроз

</details>

**Сценарий B:**
Ваш SCA-сканер нашёл критичную CVE в библиотеке `lodash`. Разработчик обновил зависимость, но забыл проверить что новая версия ломает функционал. На продакшене упал платёжный модуль.

<details>
<summary>Ответ</summary>

- **5.14** Управление недостатками — процесс обновления не включал регрессионное тестирование
- **5.12** Функциональное тестирование — не проверили критичный функционал после изменений
- **5.21** Управление изменениями — изменение прошло без оценки влияния

</details>

**Сценарий C:**
DevOps-инженер захардкодил AWS-ключ в `docker-compose.yml` для тестовой среды. Файл попал в публичный GitHub-репозиторий. Через 12 минут на вашем AWS-аккаунте запустились 40 EC2-инстансов для майнинга.

<details>
<summary>Ответ</summary>

- **5.15** Безопасность секретов — ключ в коде
- **5.4** Управление конфигурацией — файл с секретом не в `.gitignore`
- **5.3** Управление инфраструктурой — тестовая среда с production-ключами

</details>

**Сценарий D:**
Компания прошла сертификацию ФСТЭК год назад. Сейчас пришёл аудитор и просит показать, какие SAST-правила используются и как обрабатываются результаты. Никто не может найти эту информацию — человек, который настраивал сканер, уволился.

<details>
<summary>Ответ</summary>

- **5.20** Управление документацией — процессы не задокументированы
- **5.2** Инструментальная среда — конфигурация инструментов не сохранена
- **5.25** Совершенствование — не было передачи знаний

</details>

**Сценарий E:**
Вы выпустили мобильное приложение v2.0. Через полгода решили его закрыть и переключить пользователей на веб-версию. Приложение удалили из стора, сервер выключили. Через месяц выяснилось, что в базе данных на том сервере остались 50 000 записей с персональными данными пользователей.

<details>
<summary>Ответ</summary>

- **5.18** Безопасность при выводе из эксплуатации — данные не были удалены
- **5.22** Управление рисками — риски вывода не оценивались
- **5.24** Осведомлённость — пользователи не уведомлены о прекращении работы

</details>

> **Зачем:** после этого квиза вы начнёте «видеть» процессы ГОСТа в реальных ситуациях. Это гораздо ценнее, чем помнить номера разделов.

Запишите свои ответы в отдельный файл или проверяйте себя устно — это задание не требует артефакта.
