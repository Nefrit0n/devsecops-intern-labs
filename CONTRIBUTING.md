# Как контрибьютить

Спасибо за интерес к проекту! Вот как вы можете помочь.

## Что можно сделать

- Добавить задания или примеры к существующим этапам
- Улучшить воспроизводимость (reproducibility): фиксированные версии, повторяемые шаги, детерминированные примеры
- Повысить консистентность docs ↔ artifacts: синхронизировать README, задания, шаблоны, примеры и ожидаемые результаты
- Добавить автоматические проверки учебных путей: линтеры, валидацию структуры, smoke-тесты заданий и CI-проверки
- Улучшить документацию
- Добавить поддержку новых инструментов
- Расширить Juice Shop новыми уязвимостями
- Исправить ошибки

## Правила

1. Один PR = одна задача
2. Для нового модуля обязательны: `README.md`, `tasks.md`, `examples/`, `templates/`
3. Все задания должны быть привязаны к конкретным разделам ГОСТ Р 56939-2024
4. Используйте только open-source инструменты
5. Пишите на русском (основной язык репозитория)
6. Явно отделяйте материалы курса от student-generated артефактов

## Структура модуля

```
stage-N/module-name/
├── README.md      ← теория: о чём модуль, что читать, привязка к ГОСТу
├── tasks.md       ← задания + чеклист самопроверки
├── examples/      ← примеры конфигов, отчётов, скриптов
└── templates/     ← шаблоны для выполнения заданий
```

### Минимальный стандарт для новых модулей

Обязательный минимум:

- `README.md` — контекст, цели, зависимости, привязка к ГОСТу, ожидаемые результаты
- `tasks.md` — пошаговые задания, критерии приёмки, самопроверка
- `examples/` — эталонные (reference) примеры входов/выходов
- `templates/` — стартовые шаблоны, которые студент заполняет

Student-generated (не должны коммититься как готовые решения в базовый модуль):

- индивидуальные отчёты, логи и временные файлы;
- локальные артефакты выполнения (`*.tmp`, `*.cache`, дампы, локальные результаты сканирования);
- заполненные студентом шаблоны и финальные ответы, если они не оформлены как отдельные reference-примеры для курса.
