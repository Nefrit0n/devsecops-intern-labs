# docker-labs

Практикум по Docker и container security для Intern-уровня.

> Формат: 5 самостоятельных лабораторных по 20–30 минут. Каждая работа заканчивается отдельным проверяемым результатом (артефактом).

## Лабораторная 1: Жизненный цикл image и container
### Цель
Понять различие между Docker image и container через полный жизненный цикл: сборка, запуск, остановка, повторный запуск и удаление.

### Шаги выполнения
1. Соберите учебный образ один раз.
2. Посмотрите метаданные образа и его слои.
3. Запустите контейнер в фоне, проверьте состояние и логи.
4. Остановите контейнер и сравните вывод `docker ps` и `docker ps -a`.
5. Создайте короткое объяснение «image vs container» на основе наблюдений.

### Команды
```bash
docker build -t vulnerable-app:lab ./vulnerable-app

docker image ls | grep vulnerable-app
docker image inspect vulnerable-app:lab --format '{{.Id}} {{.Created}}'

docker run -d --name vuln-app-lifecycle -p 5000:5000 vulnerable-app:lab
docker ps
docker logs --tail 20 vuln-app-lifecycle

docker stop vuln-app-lifecycle
docker ps
docker ps -a
```

### Ожидаемый результат
- Видно, что **image** остаётся неизменным шаблоном, а **container** — это его исполняемый экземпляр с собственным состоянием.
- Студент понимает, почему один образ можно запускать много раз с разными именами контейнеров.

### Артефакт/результат лабораторной
Файл `reports/docker-lab1-lifecycle.md` с:
- 3–5 наблюдениями по командам `docker image ls`, `docker ps`, `docker ps -a`;
- кратким объяснением разницы image/container (4–6 предложений).

### Вопросы на понимание
- Почему контейнер может быть остановлен/удалён, а образ при этом остаётся?
- Что изменится, если запускать контейнер с флагом `--rm`?
- Почему для диагностики инцидентов полезно смотреть и `docker ps`, и `docker ps -a`?

### Критерии проверки
- [ ] Образ собран и найден локально.
- [ ] Контейнер запускался, останавливался и наблюдался в `docker ps -a`.
- [ ] В отчёте корректно объяснена разница image/container на основе фактов.

## Лабораторная 2: Воспроизводимость сборки — `latest` vs pinned
### Цель
Понять, как выбор базового образа влияет на воспроизводимость и почему `latest` рискован для стабильного процесса.

### Шаги выполнения
1. Откройте Dockerfile учебного приложения и зафиксируйте текущий `FROM`.
2. Смоделируйте два варианта: 
   - непредсказуемый (`latest`);
   - фиксированный (`major.minor` и/или digest).
3. Сравните, какой вариант лучше для повторяемых сборок в команде.
4. Сформулируйте policy-рекомендацию для учебного проекта.

### Команды
```bash
cat vulnerable-app/Dockerfile

docker pull python:latest
docker pull python:3.11-slim
docker image inspect python:latest --format '{{index .RepoDigests 0}}'
docker image inspect python:3.11-slim --format '{{index .RepoDigests 0}}'
```

### Ожидаемый результат
Студент может аргументированно объяснить, почему `latest` ухудшает предсказуемость, а pinned tag/digest улучшают повторяемость и triage проблем.

### Артефакт/результат лабораторной
Файл `reports/docker-lab2-reproducibility.md` с:
- сравнительной таблицей `latest` vs pinned (минимум 3 критерия: воспроизводимость, риск внезапных изменений, удобство расследования);
- рекомендацией, какой формат тега использовать в учебных Dockerfile.

### Вопросы на понимание
- Почему одинаковый Dockerfile с `latest` может дать разные результаты через месяц?
- Чем pinned tag отличается от pinning по digest с точки зрения контроля изменений?
- Когда допустим `latest` в учебной среде, и какие меры снижают риск?

### Критерии проверки
- [ ] Показана разница подходов `latest` и pinned.
- [ ] Рекомендация сформулирована не «по привычке», а через риски и воспроизводимость.
- [ ] В отчёте есть минимум один практический компромисс (удобство vs контроль).

## Лабораторная 3: Baseline vs Hardening в Dockerfile
### Цель
Научиться отличать базово приемлемый Dockerfile от более защищённого варианта и оценивать эффект hardening-мер.

### Шаги выполнения
1. Проанализируйте текущий Dockerfile `vulnerable-app` как baseline.
2. Найдите реализованные защитные меры (например, non-root, параметры Python runtime).
3. Предложите 3–5 улучшений hardening без переусложнения архитектуры.
4. Разделите предложения на:
   - «сделать сейчас» (быстрый выигрыш);
   - «сделать позже» (требует доп. тестирования).

### Команды
```bash
cat vulnerable-app/Dockerfile
```

### Подсказки
Можно анализировать такие аспекты:
- минимизация base image;
- запуск от non-root;
- порядок слоёв и кэширование;
- фиксация версий зависимостей;
- уменьшение surface area (только нужные пакеты/файлы).

### Ожидаемый результат
Студент понимает, что hardening — это не «магическая одна команда», а набор решений с балансом безопасности, стабильности и трудозатрат.

### Артефакт/результат лабораторной
Файл `reports/docker-lab3-hardening-review.md` с:
- блоком «Baseline (что уже хорошо)»;
- блоком «Hardening backlog» с приоритизацией (high/medium/low);
- краткой оценкой рисков, которые снижаются каждым high-priority пунктом.

### Вопросы на понимание
- Почему запуск от root повышает impact при компрометации контейнера?
- Какие hardening-меры чаще всего ломают совместимость и почему?
- Почему важно приоритизировать улучшения, а не внедрять всё сразу?

### Критерии проверки
- [ ] Названы минимум 3 сильные стороны baseline и 3 улучшения hardening.
- [ ] Есть приоритизация и аргументация по рискам.
- [ ] Предложения реалистичны для локального учебного проекта.

## Лабораторная 4: Container scanning (Trivy) и remediation thinking
### Цель
Провести сканирование контейнерного образа, интерпретировать findings и подготовить короткий remediation plan.

### Шаги выполнения
1. Используйте уже собранный образ `vulnerable-app:lab` (пересборка не нужна, если образ актуален).
2. Запустите Trivy-скан и сохраните JSON-отчёт.
3. Выделите 5 наиболее значимых находок (по severity и применимости).
4. Для каждой находки укажите действие: patch now / mitigate / accept temporarily.

### Команды
```bash
trivy image --format json -o reports/trivy-image-report.json vulnerable-app:lab
```

### Ожидаемый результат
Получен не просто «сырой» отчёт сканера, а структурированный анализ приоритетов исправления с понятной логикой.

### Артефакт/результат лабораторной
- `reports/trivy-image-report.json`
- `reports/docker-lab4-remediation-plan.md` с:
  - топ-5 findings;
  - обоснованием приоритета;
  - ближайшими шагами remediation.

### Вопросы на понимание
- Почему не каждую уязвимость нужно чинить немедленно?
- Чем отличается «уязвимость есть в образе» от «уязвимость реально эксплуатируема в данном контексте»?
- Какие факторы влияют на приоритет remediation кроме severity?

### Критерии проверки
- [ ] Отчёт Trivy сохранён в `reports/`.
- [ ] Есть разбор findings, а не только список CVE.
- [ ] В remediation plan указаны приоритет, действие и причина.

## Лабораторная 5: Локальный quality gate — warn-only vs blocking
### Цель
Понять поведение локального security pipeline в двух режимах и объяснить, когда каждый режим уместен.

### Шаги выполнения
1. Запустите полный локальный pipeline в режиме предупреждений (warn-only).
2. Запустите тот же pipeline в blocking-режиме.
3. Сравните итоговый статус и код завершения двух запусков.
4. Сформулируйте рекомендацию: какой режим использовать на этапе обучения, а какой перед «условным релизом».

### Команды
```bash
WARN_ONLY=true FAIL_ON_ANY_ERROR=false bash scanner-scripts/full_local_pipeline.sh . vulnerable-app:lab ./vulnerable-app
WARN_ONLY=false FAIL_ON_ANY_ERROR=true bash scanner-scripts/full_local_pipeline.sh . vulnerable-app:lab ./vulnerable-app
```

### Ожидаемый результат
Студент понимает, как один и тот же набор проверок может работать как обучающий feedback-цикл (warn-only) и как реальный quality gate (blocking).

### Артефакт/результат лабораторной
Файл `reports/docker-lab5-gate-comparison.md` с:
- сравнением статусов и exit code двух запусков;
- краткой интерпретацией «что изменилось и почему»;
- предложенной политикой включения blocking-режима.

### Вопросы на понимание
- Почему warn-only полезен на раннем этапе обучения, но недостаточен перед релизом?
- Как `FAIL_ON_ANY_ERROR=true` влияет на дисциплину исправлений?
- Какие риски у команды, если навсегда оставить только warn-only?

### Критерии проверки
- [ ] Зафиксированы результаты обоих запусков pipeline.
- [ ] Показано различие в поведении и итоговом статусе.
- [ ] Сформулирована осмысленная рекомендация по этапам применения режимов.
