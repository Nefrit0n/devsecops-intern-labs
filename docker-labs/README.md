# docker-labs

Лабораторные работы по Docker и container security для Intern-уровня.

## Лабораторная 1: Сборка образа
### Цель
Собрать Docker-образ учебного приложения.

### Шаги выполнения
1. Соберите образ `vulnerable-app`.
2. Проверьте, что образ появился локально.

### Команды
```bash
docker build -t vulnerable-app:lab ./vulnerable-app
docker images | grep vulnerable-app
```

### Ожидаемый результат
Образ `vulnerable-app:lab` успешно создан.

### Вопросы на понимание
- Что делает каждая ключевая инструкция в Dockerfile?
- Зачем фиксировать версии зависимостей?

### Критерии проверки
- [ ] Образ успешно собран.
- [ ] Понимаю базовые инструкции Dockerfile.

## Лабораторная 2: Разница между image и container
### Цель
Понять жизненный цикл image/container.

### Шаги выполнения
1. Запустите контейнер из собранного образа.
2. Посмотрите активные и завершённые контейнеры.

### Команды
```bash
docker run --name vuln-app-lab --rm -p 5000:5000 vulnerable-app:lab
docker ps
docker ps -a
```

### Ожидаемый результат
Можно объяснить, что image — шаблон, а container — запущенный экземпляр.

### Вопросы на понимание
- Почему контейнер может исчезнуть при `--rm`?
- Что хранится в слоях образа?

### Критерии проверки
- [ ] Могу показать container в `docker ps`.
- [ ] Могу объяснить отличие image/container.

## Лабораторная 4: Проблема latest tag
### Цель
Понять риск использования плавающих тегов.

### Шаги выполнения
1. Сравните пример с `latest` и фиксированной версией.
2. Объясните, как это влияет на воспроизводимость.

### Команды
```bash

# Более воспроизводимо:
# FROM python:3.11-slim
```

### Ожидаемый результат
Понятно, почему `latest` ухудшает предсказуемость сборки.

### Вопросы на понимание

- Когда уместно использовать digest pinning?

### Критерии проверки

- [ ] Могу предложить стабильную альтернативу.

## Лабораторная 6: Базовый container scanning через Trivy
### Цель
Провести первичный анализ образа на уязвимости.

### Шаги выполнения
1. Соберите образ приложения.
2. Запустите `trivy image`.
3. Сохраните отчёт.

### Команды
```bash

trivy image --format json -o trivy-image-report.json vulnerable-app:lab
```

### Ожидаемый результат
Получен список уязвимостей с severity.

### Вопросы на понимание
- Какие находки исправлять в первую очередь?
- Что делать, если патч пока недоступен?

### Критерии проверки
- [ ] Trivy отработал по образу.
- [ ] Есть краткий remediation plan.

## Лабораторная 7: Мини quality gate локально
### Цель
Научиться запускать локальный security pipeline в двух режимах.

### Шаги выполнения
1. Запустите pipeline в warn-only режиме.
2. Запустите pipeline в blocking режиме.
3. Сравните итоговый статус и код завершения.

### Команды
```bash
WARN_ONLY=true FAIL_ON_ANY_ERROR=false bash scanner-scripts/full_local_pipeline.sh . vulnerable-app:lab ./vulnerable-app
WARN_ONLY=false FAIL_ON_ANY_ERROR=true bash scanner-scripts/full_local_pipeline.sh . vulnerable-app:lab ./vulnerable-app
```

### Ожидаемый результат
Студент понимает разницу между информативным запуском и quality gate.

### Вопросы на понимание
- Когда уместен warn-only режим?
- Почему для quality gate нужен предсказуемый `exit 1`?

### Критерии проверки
- [ ] Виден финальный статус `PASSED/WARNINGS/FAILED`.
- [ ] Понятна логика pass/fail.
