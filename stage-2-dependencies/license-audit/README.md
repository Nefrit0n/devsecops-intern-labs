## Зачем проверять лицензии

Уязвимости — не единственный риск зависимостей. Лицензии — юридический риск:

| Лицензия | Тип | Риск для коммерческого ПО |
|----------|-----|--------------------------|
| MIT, BSD, Apache-2.0 | Permissive | Минимальный — используй свободно |
| LGPL-2.1, MPL-2.0 | Weak copyleft | Средний — нужно соблюдать условия |
| GPL-2.0, GPL-3.0 | Strong copyleft | Высокий — ваш код тоже станет GPL |
| AGPL-3.0 | Network copyleft | Максимальный — даже SaaS-использование обязывает |
| Нет лицензии | Unlicensed | Неопределённый — формально нельзя использовать |

**Пример:** если ваш коммерческий продукт использует GPL-3.0 библиотеку и вы распространяете бинарник — вы обязаны открыть исходный код. AGPL-3.0 ещё строже: даже если вы не распространяете, а просто запускаете как SaaS — обязательства те же.

ГОСТ напрямую не говорит «проверяйте лицензии», но процесс 5.23 (управление поставщиками) подразумевает контроль всех аспектов сторонних компонентов, включая юридические.

## Три инструмента — три глубины

| Инструмент | Где ищет лицензии | Глубина |
|------------|-------------------|---------|
| **Trivy license** | package.json, lockfiles | Быстро, но только manifest |
| **ScanCode Toolkit** | Каждый файл в проекте | Глубоко: находит заголовки внутри .js/.py |
| **license_finder** | Manifest + policy engine | Whitelist/blacklist + CI-интеграция |
