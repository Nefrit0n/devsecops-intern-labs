## Что такое SCA (и чем отличается от SAST)

| | SAST (этап 1) | SCA (этап 2) |
|--|---------------|-------------|
| Что анализирует | Ваш код | Чужой код (зависимости) |
| Что ищет | Баги в логике, небезопасные паттерны | Известные CVE в библиотеках |
| Источник данных | Исходный код | NVD, GitHub Advisory, OSV |
| Пример finding | «SQL injection в строке 42» | «lodash 4.17.15 — CVE-2021-23337 (prototype pollution)» |

SCA не читает код зависимости — он сверяет **название + версию** с базами уязвимостей. Поэтому SCA невероятно быстрый, но может давать false positives: CVE есть в библиотеке, но ваш код не вызывает уязвимую функцию.

## Direct vs Transitive зависимости

```
ваш package.json
  ├── express@4.18.2          ← direct (вы поставили)
  │   ├── body-parser@1.20.1  ← transitive (приехала с express)
  │   │   └── qs@6.11.0       ← transitive 2-го уровня
  │   └── cookie@0.5.0
  └── jsonwebtoken@9.0.0      ← direct
      └── jws@3.2.2           ← transitive
          └── jwa@1.4.1
```

У Juice Shop ~50 direct зависимостей и ~1500 transitive. CVE чаще всего прячется в transitive — вы её не ставили и не знаете о ней.

## Четыре инструмента — четыре подхода

| Инструмент | Как работает | Аналогия |
|------------|-------------|----------|
| **npm audit** | Читает package-lock.json, сверяет с GitHub Advisory | «Быстрая проверка на входе» |
| **Trivy** | Сканирует lockfiles + контейнеры + несколько БД | «Универсальный досмотр» |
| **OWASP Dep-Check** | CPE matching против NVD | «Дотошный аудитор с папкой» |
| **Grype** | Сканирует SBOM (от Syft) | «Специалист, работающий по описи» |
